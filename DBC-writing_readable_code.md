    <article class="markdown-body entry-content" itemprop="text"><h1><a id="user-content-writing-readable-code" class="anchor" href="#writing-readable-code" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Writing Readable Code</h1>
<p>Written by <a href="https://github.com/toreyhickman">Torey Hickman</a>.</p>
<h2><a id="user-content-introduction" class="anchor" href="#introduction" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Introduction</h2>
<p>When you write code, you have two audiences:  the computer and people.</p>
<p>The computer will parse and execute your code, and so your code needs to follow certain guidelines.  For example, when you define a method, you begin with <code>def</code> and close with <code>end</code>.  If you don't follow that pattern, the computer won't be able to execute the code.  If you try to run code with improper syntax, you get an error:</p>
<pre><code>example.rb:4: syntax error, unexpected end-of-input, expecting keyword_end
</code></pre>
<p>As long as the syntax is correct, your code will run.  The computer doesn't care what you name variables or methods, how many methods you chain together, or anything else.</p>
<p>However, people do care.  The "Ruby way" is to write code that is easily understandable for programmers who might encounter your code:  your pair, a teammate, a programmer who picks up your code base in the future, even your future self.</p>
<h2><a id="user-content-readable-code-best-practices" class="anchor" href="#readable-code-best-practices" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Readable Code Best Practices</h2>
<h3><a id="user-content-style" class="anchor" href="#style" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Style</h3>
<p>There is a <a href="https://github.com/styleguide/ruby">Ruby Style Guide</a> on GitHub that describes how your code should be written.  It includes guidelines on indentation and white space, delimiting blocks with <code>{...}</code> or <code>do ... end</code>, naming, etc.  Familiarize yourself with these guidelies and apply them to your code.</p>
<p>Pay particular attention to indentation.  Indentation will make your code much easier to read:  what lines belong to which methods, when does an <code>if</code> statement end, which lines are with a block, etc.  If code is not indented properly, it's much more difficult for a human to parse (i.e., read).</p>
<h3><a id="user-content-variable-and-method-naming" class="anchor" href="#variable-and-method-naming" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Variable and Method Naming</h3>
<p>Names should be descriptive.  A variable name should inform readers of the value assigned to the variable.  Methods should be named the same way:  inform readers what the method does ... or at least what it returns.</p>
<p>Variables with names like <code>x</code> might make sense in the moment you're writing code, but a few days later, you might have to ask yourself, "Now, what was 'x'?"  And, someone unfamiliar with your code would have no clue.  In addition, variable names like <code>string</code> or <code>array</code> pass on information about the type of object assigned to the variable, but not what the object represents.</p>
<div class="highlight highlight-source-ruby"><pre><span class="pl-c"><span class="pl-c">#</span> poor naming choices</span>
<span class="pl-k">require</span> <span class="pl-s"><span class="pl-pds">'</span>prime<span class="pl-pds">'</span></span>

<span class="pl-k">def</span> <span class="pl-en">p_factors</span>(<span class="pl-smi">x</span>)
  <span class="pl-k">return</span> [x] <span class="pl-k">if</span> x.prime?

  pf <span class="pl-k">=</span> p_factor(x)
  n <span class="pl-k">=</span> x <span class="pl-k">/</span> pf

  [pf] <span class="pl-k">+</span> p_factors(n)
<span class="pl-k">end</span>

<span class="pl-k">def</span> <span class="pl-en">p_factor</span>(<span class="pl-smi">x</span>)
  pos_factors(x).find { |<span class="pl-smi">p</span>| p.prime? <span class="pl-k">&amp;&amp;</span> x <span class="pl-k">%</span> p <span class="pl-k">==</span> <span class="pl-c1">0</span> }
<span class="pl-k">end</span>

<span class="pl-k">def</span> <span class="pl-en">pos_factors</span>(<span class="pl-smi">x</span>)
  (<span class="pl-c1">2</span>..(x <span class="pl-k">/</span> <span class="pl-c1">2</span>))
<span class="pl-k">end</span>


<span class="pl-c"><span class="pl-c">#</span> better naming choices</span>
<span class="pl-k">require</span> <span class="pl-s"><span class="pl-pds">'</span>prime<span class="pl-pds">'</span></span>

<span class="pl-k">def</span> <span class="pl-en">prime_factors</span>(<span class="pl-smi">number</span>)
  <span class="pl-k">return</span> [number] <span class="pl-k">if</span> number.prime?

  prime_factor <span class="pl-k">=</span> find_prime_factor_of(number)
  next_number_to_check <span class="pl-k">=</span> number <span class="pl-k">/</span> prime_factor

  [prime_factor] <span class="pl-k">+</span> prime_factors(next_number_to_check)
<span class="pl-k">end</span>

<span class="pl-k">def</span> <span class="pl-en">find_prime_factor_of</span>(<span class="pl-smi">number</span>)
  possible_factors_of(number).find <span class="pl-k">do </span>|<span class="pl-smi">possibility</span>|
    possibility.prime? <span class="pl-k">&amp;&amp;</span> number <span class="pl-k">%</span> possibility <span class="pl-k">==</span> <span class="pl-c1">0</span>
  <span class="pl-k">end</span>
<span class="pl-k">end</span>

<span class="pl-k">def</span> <span class="pl-en">possible_factors_of</span>(<span class="pl-smi">number</span>)
  (<span class="pl-c1">2</span>..(number <span class="pl-k">/</span> <span class="pl-c1">2</span>))
<span class="pl-k">end</span></pre></div>
<h3><a id="user-content-method-chaining" class="anchor" href="#method-chaining" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Method Chaining</h3>
<p>Chaining method calls together can make code more concise and readable:</p>
<div class="highlight highlight-source-ruby"><pre><span class="pl-k">def</span> <span class="pl-en">doubled_evens</span>(<span class="pl-smi">numbers</span>)
  numbers.select(<span class="pl-k">&amp;</span><span class="pl-c1">:even?</span>).map { |<span class="pl-smi">number</span>| number <span class="pl-k">*</span> <span class="pl-c1">2</span> }
<span class="pl-k">end</span>

<p>In the code above, the body of the <code>doubled_evens</code> method shows an example of method chaining.  Notice that the data type does not change.  The method accepts an array argument and calls <code>#select</code> on that array, returning another array.  <code>#map</code> is called on this returned array, returning yet another array.</p>
<p>Be wary of hiding a change of data type (e.g., array to string) within a chain of method calls.</p>
<p>Also, remember that you are attempting to write readable code.  Less lines of code is generally a good thing, but not if it makes the code harder to understand. The computer will chain a dozen method calls without a problem, but it will be difficult for a person to read.</p>
